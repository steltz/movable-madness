## 1. Nx Workspace Setup with pnpm

### Decision: Use `ts` preset with pnpm package manager

**Rationale**: The `ts` (TypeScript) preset provides maximum flexibility for custom configuration while avoiding unwanted defaults (ESLint/Prettier). The integrated monorepo mode creates proper `apps/` and `libs/` separation with shared `tsconfig.base.json`.

**Alternatives Considered**:
- `react-monorepo` preset: Rejected because it installs ESLint by default
- `empty` preset: Rejected because it requires more manual setup
- `npm` preset: Rejected because we need pnpm

### Workspace Creation Command

```bash
pnpm create nx-workspace@latest my-workspace --pm=pnpm --preset=ts --nxCloud=skip
```

### Required Configuration Files

**pnpm-workspace.yaml**:
```yaml
packages:
  - 'apps/*'
  - 'libs/*'
```

**nx.json** (key settings):
```json
{
  "$schema": "./node_modules/nx/schemas/nx-schema.json",
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "cache": true
    },
    "serve": {
      "cache": true
    }
  },
  "defaultBase": "main"
}
```

### Removing ESLint/Prettier

If generated by Nx generators:
```bash
pnpm remove -w eslint prettier @nx/eslint @nx/eslint-plugin
rm -f .eslintrc.json .prettierrc .prettierignore eslint.config.js
```

Remove from `nx.json` plugins array if present.

## 2. Biome Configuration

### Decision: Single root biome.json with workspace-wide rules

**Rationale**: Biome v1.8+ has native monorepo support. A single root configuration simplifies maintenance and ensures consistency. Biome is 10-25x faster than ESLint + Prettier combined.

**Alternatives Considered**:
- Per-project biome.json files: Rejected for simplicity; can add later if needed
- @gitopslovers/nx-biome plugin: Rejected; native Biome is sufficient for workspace-wide linting

### Recommended biome.json

```json
{
  "$schema": "https://biomejs.dev/schemas/1.9.0/schema.json",
  "vcs": {
    "enabled": true,
    "clientKind": "git",
    "useIgnoreFile": true,
    "defaultBranch": "main"
  },
  "files": {
    "ignoreUnknown": false,
    "ignore": ["dist", "node_modules", ".nx", "coverage"]
  },
  "formatter": {
    "enabled": true,
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100
  },
  "organizeImports": {
    "enabled": true
  },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true,
      "correctness": {
        "noUnusedImports": "error",
        "noUnusedVariables": "warn"
      },
      "style": {
        "useImportType": "warn",
        "useConst": "error"
      },
      "suspicious": {
        "noExplicitAny": "warn"
      }
    }
  },
  "javascript": {
    "parser": {
      "unsafeParameterDecoratorsEnabled": true
    },
    "formatter": {
      "quoteStyle": "single",
      "trailingCommas": "es5",
      "semicolons": "always"
    }
  },
  "overrides": [
    {
      "include": ["apps/api/**/*.ts", "apps/functions/**/*.ts"],
      "linter": {
        "rules": {
          "style": {
            "useImportType": "off"
          }
        }
      }
    }
  ]
}
```

### CI Integration

```bash
biome ci .
```

Returns non-zero exit code on violations. Use `--changed --since=origin/main` for PR checks.

## 3. Firebase Functions 2nd Gen

### Decision: Manual Nx setup with @nx/esbuild executor

**Rationale**: Direct control over build configuration without third-party plugin dependencies. esbuild provides fast bundling with proper package.json generation for Firebase deployment.

**Alternatives Considered**:
- @simondotm/nx-firebase plugin: Viable but adds external dependency
- @nx/node executor: Less optimized for Firebase Functions deployment structure

### Project Configuration

**apps/functions/project.json**:
```json
{
  "name": "functions",
  "sourceRoot": "apps/functions/src",
  "projectType": "application",
  "targets": {
    "build": {
      "executor": "@nx/esbuild:esbuild",
      "outputs": ["{options.outputPath}"],
      "options": {
        "main": "apps/functions/src/main.ts",
        "outputPath": "dist/apps/functions",
        "tsConfig": "apps/functions/tsconfig.app.json",
        "platform": "node",
        "format": ["cjs"],
        "bundle": true,
        "generatePackageJson": true,
        "thirdParty": false,
        "esbuildOptions": {
          "outExtension": { ".js": ".js" }
        }
      }
    }
  }
}
```

### Main.ts Structure (2nd Gen)

```typescript
import { onRequest } from 'firebase-functions/v2/https';
import * as admin from 'firebase-admin';
import type { ApiResponse } from '@workspace/shared-types';

admin.initializeApp();

export const helloWorld = onRequest((req, res) => {
  const response: ApiResponse = {
    success: true,
    message: 'Hello from Firebase Functions!',
    timestamp: new Date().toISOString()
  };
  res.json(response);
});
```

### Package.json for dist (auto-generated)

The `generatePackageJson: true` option creates:
```json
{
  "name": "functions",
  "main": "main.js",
  "dependencies": {
    "firebase-admin": "^12.x.x",
    "firebase-functions": "^5.x.x"
  }
}
```

### Firebase.json Configuration

```json
{
  "functions": {
    "source": "dist/apps/functions",
    "runtime": "nodejs20",
    "predeploy": ["pnpm nx build functions"]
  },
  "hosting": {
    "public": "dist/apps/web",
    "rewrites": [
      {
        "source": "/api/**",
        "run": {
          "serviceId": "api",
          "region": "us-central1"
        }
      },
      {
        "source": "**",
        "destination": "/index.html"
      }
    ]
  }
}
```

## 4. Docker for Cloud Run

### Decision: Multi-stage Dockerfile with node:22-slim base

**Rationale**: Node.js 22 is current LTS with performance improvements. Slim variant provides good security (900+ fewer vulnerabilities than full images) while maintaining glibc compatibility.

**Alternatives Considered**:
- Alpine: Rejected due to musl/glibc compatibility issues with native modules
- Node 20: Viable but 22 has better performance and longer support window

### Recommended Dockerfile

```dockerfile
# Stage 1: Dependencies
FROM node:22-slim AS deps
RUN corepack enable pnpm
WORKDIR /app

COPY pnpm-lock.yaml pnpm-workspace.yaml ./
COPY apps/api/package.json ./apps/api/
COPY libs/shared-types/package.json ./libs/shared-types/

RUN --mount=type=cache,id=pnpm-store,target=/root/.pnpm-store \
    pnpm fetch --frozen-lockfile

# Stage 2: Builder
FROM node:22-slim AS builder
RUN corepack enable pnpm
WORKDIR /app

COPY --from=deps /root/.pnpm-store /root/.pnpm-store
COPY . .

RUN --mount=type=cache,id=pnpm-store,target=/root/.pnpm-store \
    pnpm install --frozen-lockfile && \
    pnpm nx build api --prod

# Stage 3: Production
FROM node:22-slim AS runner

ENV NODE_ENV=production
ENV PORT=8080

RUN groupadd -r nodejs && useradd -r -g nodejs nodejs

WORKDIR /app

COPY --from=builder /app/dist/apps/api ./
COPY --from=builder /app/node_modules ./node_modules

USER nodejs
EXPOSE 8080

CMD ["node", "main.js"]
```

### Cloud Run Optimizations

1. **Direct node invocation**: Use `node main.js` not `npm start`
2. **Memory limits**: Set `NODE_OPTIONS="--max-old-space-size=256"`
3. **Port**: Cloud Run expects port 8080
4. **Non-root user**: Required for security best practices
5. **Startup CPU boost**: Enable in Cloud Run console for 50% faster cold starts

## 5. Version Recommendations

| Package | Version | Notes |
|---------|---------|-------|
| Nx | ^19.8.x | Latest stable with Project Crystal |
| @nx/react | ^19.8.x | Match Nx version |
| @nx/nest | ^19.8.x | Match Nx version |
| @nx/esbuild | ^19.8.x | Match Nx version |
| React | ^18.3.x | Latest stable |
| Vite | ^5.4.x | Latest stable |
| NestJS | ^10.4.x | Latest stable |
| Biome | ^1.9.x | Latest with monorepo support |
| firebase-admin | ^12.x.x | Compatible with functions v5 |
| firebase-functions | ^5.1.x | 2nd Gen support |
| Node.js | 20 or 22 | LTS versions |
| pnpm | ^9.x.x | Latest stable |
| TypeScript | ^5.5.x | Latest stable |

### Compatibility Notes

- **firebase-admin 13.x is NOT compatible** with firebase-functions 6.x (peer dependency issue)
- Keep all `@nx/*` packages at the same major.minor version
- NestJS requires `unsafeParameterDecoratorsEnabled: true` in Biome for decorators
- Use `useImportType: off` for NestJS files to avoid DI issues
